## 1. Database Schema Overview

### Tables Involved

| Table | Purpose |
|---|---|
| `enrollments` | Created on every course purchase (single or inside bundle) |
| `bundle_enrollments` | Created when student buys a bundle |
| `faculty_transactions` | Created only at payout time — tracks sales + transfers |
| `profiles` | Faculty profile — holds per-faculty `commission_rate` (nullable) |
| `platform_settings` | Default commission % (fallback), other global config |

### Key Columns

**enrollments**
```
id                    → primary key
student_id            → who bought
course_id             → which course
faculty_id            → which faculty earns
amount_paid           → total paid by student (in paise)
gst_amount            → GST portion (in paise)
course_price          → original course price
is_bundle_enrollment  → true if came from bundle purchase
bundle_enrollment_id  → links to bundle_enrollments if is_bundle_enrollment=true
payment_id            → razorpay/payment gateway id
payment_status        → SUCCESS | PENDING | FAILED (default SUCCESS)
                        only SUCCESS rows count toward revenue & payouts
```

**bundle_enrollments**
```
id          → primary key
student_id  → who bought
bundle_id   → which bundle
faculty_id  → which faculty owns the bundle
amount_paid → total bundle amount paid (in paise)
payment_id  → razorpay/payment gateway id
payment_status → SUCCESS | PENDING | FAILED (default SUCCESS)
                 only SUCCESS rows count toward revenue & payouts
```

**faculty_transactions**
```
id                    → primary key
faculty_id            → which faculty
transaction_id        → auto generated by trigger
type                  → COURSE_SALE | BUNDLE_SALE | PAYOUT | PLATFORM_FEE | PLATFORM_EARNING | REFUND
status                → PENDING | SUCCESS | FAILED
amount                → faculty share (COURSE_SALE/BUNDLE_SALE/PAYOUT) OR commission (PLATFORM_FEE per enrollment, PLATFORM_EARNING run total)
gross_amount          → base amount before commission deduction
gst_amount            → GST amount for reference
commission_percent    → % rate used at time of payout (audit trail)
enrollment_id         → filled for COURSE_SALE rows
bundle_enrollment_id  → filled for BUNDLE_SALE rows
payout_id             → points to PAYOUT transaction_id (for COURSE_SALE, BUNDLE_SALE, PLATFORM_FEE, PLATFORM_EARNING rows)
payment_id            → payment gateway/transfer reference — ONLY on PAYOUT and PLATFORM_EARNING rows (null for all other types, enforced by check constraint)
payout_time_period    → enrollment period this run settles, e.g. "June 2026" (set on EVERY row of the run)
transacted_at         → when this transaction happened
```

**profiles** (faculty)
```
id               → faculty user id
commission_rate  → per-faculty commission % (numeric, nullable)
                   null = use platform default
```

**platform_settings**
```
key    → 'default_commission_percent'
value  → '20'  (fallback — used only when profiles.commission_rate is null)
```

## 2. Flow Overview

```
SINGLE COURSE:                        BUNDLE:
Student buys course                   Student buys bundle
        ↓                                     ↓
enrollments row created               bundle_enrollments row created
  is_bundle_enrollment = false          faculty_id, amount_paid stored
  amount_paid, gst_amount stored              ↓
        ↓                             [trigger fires automatically]
   [Nothing else]                     create_enrollments_on_bundle_purchase
                                              ↓
                                      N enrollments rows created
                                        is_bundle_enrollment = true
                                        bundle_enrollment_id = filled
                                        amount_paid = 0 (bundle price is on bundle_enrollments)
                                        ↓
                                       [Nothing else]

═════════════════════════════════════════════════
                  MONTHLY PAYOUT
═════════════════════════════════════════════════

Admin clicks "Process Payout" for a faculty
        ↓
Get commission % for this faculty
(profiles.commission_rate → fallback: platform_settings default)
        ↓
Find unpaid single enrollments
(enrollments with no COURSE_SALE in faculty_transactions)
        ↓
Find unpaid bundle enrollments
(bundle_enrollments with no BUNDLE_SALE in faculty_transactions)
        ↓
Calculate faculty share per enrollment
        ↓
Create COURSE_SALE row per single enrollment
Create BUNDLE_SALE row per bundle enrollment
Create PLATFORM_FEE row per enrollment (admin commission record)
Create ONE PAYOUT row (total transfer amount)
Create ONE PLATFORM_EARNING row (total platform commission for the run)
        ↓
Link all rows via payout_id
```

## 3. Track 1 — Single Course Purchase

### What happens
```
Student pays ₹1,180 for a course (₹1,000 + ₹180 GST)

enrollments row:
  amount_paid          = 118000  (paise)
  gst_amount           = 18000   (paise)
  is_bundle_enrollment = false
  bundle_enrollment_id = null
  faculty_id           = faculty-A
```

### At payout time
```
base_amount    = amount_paid - gst_amount  = 100000 paise (₹1,000)
commission     = base_amount * 20%         =  20000 paise (₹200)
faculty_share  = base_amount - commission  =  80000 paise (₹800)
```

### Rows created in faculty_transactions
```
Row 1 → COURSE_SALE:
  type               = COURSE_SALE
  amount             = 80000        ← faculty gets this
  gross_amount       = 100000       ← base after GST
  gst_amount         = 18000        ← GST excluded
  commission_percent = 20.00
  enrollment_id      = enroll-001
  payout_id          = TXN-PAY-001  ← links to PAYOUT row

Row 2 → PLATFORM_FEE:
  type               = PLATFORM_FEE
  amount             = 20000        ← admin keeps this
  gross_amount       = 100000
  gst_amount         = 18000
  commission_percent = 20.00
  enrollment_id      = enroll-001
  payout_id          = TXN-PAY-001
```

## 4. Track 2 — Bundle Purchase

### What happens
```
Student pays ₹2,500 for a bundle (3 courses inside)

bundle_enrollments row:
  amount_paid = 250000  (paise)
  faculty_id  = faculty-A
  bundle_id   = bundle-001

[trigger fires]

3 enrollments rows auto created:
  is_bundle_enrollment = true
  bundle_enrollment_id = bundle-enroll-001
  amount_paid          = 0   ← individual rows have 0, price is on bundle_enrollments
  faculty_id           = faculty-A
```

### At payout time
```
base_amount    = bundle amount_paid         = 250000 paise (₹2,500)
commission     = base_amount * 20%          =  50000 paise (₹500)
faculty_share  = base_amount - commission   = 200000 paise (₹2,000)

NOTE: Calculate commission on bundle_enrollments.amount_paid
      NOT on individual enrollment rows (those have amount_paid = 0)
```

### Rows created in faculty_transactions
```
Row 1 → BUNDLE_SALE:
  type                 = BUNDLE_SALE
  amount               = 200000           ← faculty gets this
  gross_amount         = 250000
  gst_amount           = 0                ← bundle may handle GST differently
  commission_percent   = 20.00
  bundle_enrollment_id = bundle-enroll-001
  enrollment_id        = null
  payout_id            = TXN-PAY-001

Row 2 → PLATFORM_FEE:
  type                 = PLATFORM_FEE
  amount               = 50000            ← admin keeps this
  gross_amount         = 250000
  commission_percent   = 20.00
  bundle_enrollment_id = bundle-enroll-001
  payout_id            = TXN-PAY-001
```

## 5. Monthly Payout Process

> **Period scoping:** a payout run settles only the **previous calendar month's**
> enrollments (run in July → settles all of June). The month label is stored on
> every row as `payout_time_period` (e.g. "June 2026"). Enrollments from the current
> month are left for next month's run.


### Full Example

```
Faculty-A this month:
  Single course: 2 students → enroll-001 (₹1,000), enroll-002 (₹1,000)
  Bundle:        1 student  → bundle-enroll-001 (₹2,500)

Commission: 20% (faculty-A's resolved rate: profiles.commission_rate, or platform default if null)

Calculations:
  enroll-001:        base=₹1,000  commission=₹200   faculty=₹800
  enroll-002:        base=₹1,000  commission=₹200   faculty=₹800
  bundle-enroll-001: base=₹2,500  commission=₹500   faculty=₹2,000

  Total faculty payout = ₹800 + ₹800 + ₹2,000 = ₹3,600
```

### Rows created in faculty_transactions

```
┌──────────────┬────────────────┬────────┬──────────────────┬───────────────┬──────────────────────┬─────────────┐
│ faculty_id   │ transaction_id │ amount │ type             │ enrollment_id │ bundle_enrollment_id │ payout_id   │
├──────────────┼────────────────┼────────┼──────────────────┼───────────────┼──────────────────────┼─────────────┤
│ faculty-A    │ TXN-0001       │ 80000  │ COURSE_SALE      │ enroll-001    │ null                 │ TXN-PAY-001 │
│ faculty-A    │ TXN-0002       │ 20000  │ PLATFORM_FEE     │ enroll-001    │ null                 │ TXN-PAY-001 │
│ faculty-A    │ TXN-0003       │ 80000  │ COURSE_SALE      │ enroll-002    │ null                 │ TXN-PAY-001 │
│ faculty-A    │ TXN-0004       │ 20000  │ PLATFORM_FEE     │ enroll-002    │ null                 │ TXN-PAY-001 │
│ faculty-A    │ TXN-0005       │ 200000 │ BUNDLE_SALE      │ null          │ bundle-enroll-001    │ TXN-PAY-001 │
│ faculty-A    │ TXN-0006       │ 50000  │ PLATFORM_FEE     │ null          │ bundle-enroll-001    │ TXN-PAY-001 │
│ faculty-A    │ TXN-PAY-001    │ 360000 │ PAYOUT           │ null          │ null                 │ TXN-PAY-001 │
│ faculty-A    │ TXN-0007       │ 90000  │ PLATFORM_EARNING │ null          │ null                 │ TXN-PAY-001 │
└──────────────┴────────────────┴────────┴──────────────────┴───────────────┴──────────────────────┴─────────────┘

amounts in paise · every row's transaction_id is unique (auto generated by trigger)
The PAYOUT row's own transaction_id (TXN-PAY-001) is what every other row's payout_id points to

Sanity check: PAYOUT (360000) + PLATFORM_EARNING (90000) = 450000 = total base amount of all sales
PLATFORM_EARNING amount = SUM of all PLATFORM_FEE rows in the run (20000 + 20000 + 50000 = 90000)
```

### payout_id linking
```
PAYOUT row transaction_id  = "TXN-PAY-001"
All other rows payout_id   = "TXN-PAY-001"
(incl. the PLATFORM_EARNING summary row)

→ Query "show everything in March payout" = filter payout_id = "TXN-PAY-001"
```

### Real Payout Run Example (2 single-course sales, 16% commission)

> A concrete run captured from the database — one "Process Payout" click for a
> faculty with 2 unpaid course sales. Commission resolved to 16%
> (`profiles.commission_rate`, else platform default).
>
> Per enrollment: `base = amount_paid − gst_amount`,
> `commission = round(base × 16%)`, `faculty share = base − commission`.

| transaction_id | type | enrollment | gross (base) | gst | amount | meaning |
|---|---|---|---|---|---|---|
| …000015 | `PAYOUT` | — | 7,909 | 791 | **6,643** | total transferred to faculty (anchor: own transaction_id = payout_id) |
| …000016 | `COURSE_SALE` | enr-A (paid ₹4,500) | 4,091 | 409 | 3,436 | faculty share |
| …000018 | `PLATFORM_FEE` | enr-A | 4,091 | 409 | 655 | platform's 16% |
| …000017 | `COURSE_SALE` | enr-B (paid ₹4,200) | 3,818 | 382 | 3,207 | faculty share |
| …000019 | `PLATFORM_FEE` | enr-B | 3,818 | 382 | 611 | platform's 16% |
| …000020 | `PLATFORM_EARNING` | — | 7,909 | 791 | **1,266** | run's total commission (Σ PLATFORM_FEE) |

```
Calculations:
  enr-A: base = 4500 − 409 = 4091   commission = round(4091 × 16%) = 655   faculty = 3436
  enr-B: base = 4200 − 382 = 3818   commission = round(3818 × 16%) = 611   faculty = 3207

Roll-ups:
  PAYOUT amount           = 3436 + 3207 = 6643   (Σ COURSE_SALE)
  PLATFORM_EARNING amount = 655  + 611  = 1266   (Σ PLATFORM_FEE)
  gross_amount (both)     = 4091 + 3818 = 7909
  gst_amount (both)       = 409  + 382  = 791

Reconciliation:
  PAYOUT (6643) + PLATFORM_EARNING (1266) = 7909 = total base ✓

Where the students' ₹8,700 went:
  ₹791 GST → government   ₹6,643 → faculty   ₹1,266 → platform
```

**Things this example makes concrete:**
- `payment_id` is set ONLY on `PAYOUT` and `PLATFORM_EARNING` (the money-movement rows);
  the per-enrollment `COURSE_SALE` / `PLATFORM_FEE` detail rows have `payment_id = null`.
- The `PAYOUT` row's `payout_id` is `null` — it IS the anchor; every other row's
  `payout_id` points to the PAYOUT's `transaction_id` (`…000015`).
- `transaction_id`s are unique & sequential; the PAYOUT's is simply the first one
  inserted (`…000015`), which is why it became the run's `payout_id`.
- Commission is rounded **per enrollment**, then summed — not rounded once on the total.

## 6. Pending Payout Calculation

### Logic
```
Pending Payout =
  unpaid single enrollments (no COURSE_SALE transaction yet) after commission
  +
  unpaid bundle enrollments (no BUNDLE_SALE transaction yet) after commission

"Unpaid" = enrollment exists in enrollments/bundle_enrollments
           but has NO matching row in faculty_transactions
```

### Admin Financial Management cards (implemented)

| Card | Source | Formula |
|---|---|---|
| Total Revenue | `faculty_transactions` PLATFORM_EARNING + unpaid commission | realized platform commission + admin commission pending on unpaid enrollments |
| Today's Revenue | today's SUCCESS enrollments + bundles | commission share of today's sales: (amount_paid − GST) × rate% |
| Pending Payout | Unpaid enrollments only | faculty share: (amount_paid − GST) × (1 − rate%) |

Rules for all three cards:
- commission rate resolved per faculty (profiles.commission_rate → platform default)
- only rows with payment_status = 'SUCCESS' are counted
- each unpaid sale splits exactly: admin commission (→ Total Revenue) + faculty share (→ Pending Payout)

## 7. Commission Settings

Each faculty can have their own commission rate stored on their profile.
The platform-wide setting is only a fallback.

### Resolution order (at payout time)
```
1. profiles.commission_rate   → per-faculty rate (used when NOT null)
2. platform_settings          → 'default_commission_percent' fallback

The resolved % is recorded in commission_percent on every transaction row
→ future rate changes never alter historical records
```

```sql
-- Per-faculty rate (null = use platform default)
select commission_rate from profiles
where id = '<faculty-uuid>';

-- Platform default (fallback)
select value from platform_settings
where key = 'default_commission_percent';
-- returns '20'
```

### Changing commission rates
```sql
-- Set a custom rate for one faculty (e.g. 15%)
update profiles
set commission_rate = 15
where id = '<faculty-uuid>';

-- Put a faculty back on the platform default
update profiles
set commission_rate = null
where id = '<faculty-uuid>';

-- Change the platform default (affects only faculties with null commission_rate)
update platform_settings
set value = '15'
where key = 'default_commission_percent';
-- Takes effect from next payout run
```

enrollment-1: paid ₹1,180  (base ₹1,000 + GST ₹180)
enrollment-2: paid ₹2,360  (base ₹2,000 + GST ₹360)
enrollment-3: paid ₹590    (base ₹500  + GST ₹90)
enrollment-4: paid ₹1,180  (base ₹1,000 + GST ₹180)
enrollment-5: paid ₹3,540  (base ₹3,000 + GST ₹540)


gross_amount  gst_amount  commission%   amount
                     ─────────────────────────────────────────────────
COURSE_SALE  enr-1 →    1000          180         20.00        800
PLATFORM_FEE enr-1 →    1000          180         20.00        200

COURSE_SALE  enr-2 →    2000          360         20.00       1600
PLATFORM_FEE enr-2 →    2000          360         20.00        400

COURSE_SALE  enr-3 →     500           90         20.00        400
PLATFORM_FEE enr-3 →     500           90         20.00        100

COURSE_SALE  enr-4 →    1000          180         20.00        800
PLATFORM_FEE enr-4 →    1000          180         20.00        200

COURSE_SALE  enr-5 →    3000          540         20.00       2400
PLATFORM_FEE enr-5 →    3000          540         20.00        600



gross_amount = 1000 + 2000 + 500 + 1000 + 3000 = 7500  ← total base
gst_amount   = 180  + 360  + 90  + 180  + 540  = 1350  ← total GST
amount       = 800  + 1600 + 400 + 800  + 2400  = 6000  ← total transferred to faculty

                         gross_amount  gst_amount  commission%   amount
                         ─────────────────────────────────────────────────
PAYOUT row           →      7500         1350         20.00       6000
PLATFORM_EARNING row →      7500         1350         20.00       1500



Verify it makes sense

Total student paid  = 7500 + 1350      = ₹8,850   (gross + GST)
Platform keeps      = 7500 × 20%       = ₹1,500   (commission)
Faculty gets        = 7500 - 1500      = ₹6,000   ✅ matches PAYOUT amount
GST goes to govt    = ₹1,350           (not anyone's income)

Cross check:
  SUM of all COURSE_SALE amounts  = 800+1600+400+800+2400 = 6000 ✅ = PAYOUT amount
  SUM of all PLATFORM_FEE amounts = 200+400+100+200+600   = 1500 ✅ = PLATFORM_EARNING amount
  6000 + 1500                     = 7500 = gross_amount   ✅
